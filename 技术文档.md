### 3.2 Oracle: Pyth Network (★★★★★)

**Why Pyth**:
- 53% of Sui's Total Value Secured (~$700M)
- 8+ protocol integrations (Cetus, NAVI, Suilend, Scallop, etc.)
- Pull Oracle: on-demand, ~1 MIST per update (~$0.000000001)
- 500+ price pairs

**Integration Pattern**:

```
Client (TypeScript)                    On-chain (Move)
+--------------------+                +---------------------------+
| 1. Fetch update    |                | oracle.move               |
|    from Hermes     |   --PTB-->     |                           |
| 2. Build PTB:      |                | public fun get_price(     |
|    update + cycle() |                |   clock: &Clock,          |
| 3. Submit tx       |                |   info: &PriceInfoObject  |
+--------------------+                | ): (u64, u64)             |
                                      +---------------------------+
```

**Critical Rules**:
- ⚠️ NEVER call `pyth::update_single_price_feed` from Move contracts
- Price updates MUST come from client-side PTBs
- Set `max_age = 60` seconds for freshness
- Pyth package addresses change on upgrade; resolve client-side

### 3.3 DEX: Cetus CLMM (Primary) + Aftermath (Router)

**Cetus**: Largest Sui DEX, ~$43M daily volume, open-source Move interface.

**⚠️ SECURITY NOTE — Cetus $223M Exploit (May 2025)**:
- **Root Cause**: Integer overflow in `integer-mate` math library
- **NOT a Move language flaw** — it was a dependency bug
- **Status**: Protocol recovered; $162M funds recovered
- **Our Mitigations**:
  1. Self-built `math.move` — NO 3rd-party math dependencies
  2. Return value validation on ALL external calls
  3. Per-transaction amount caps
  4. Aftermath Router for DEX risk diversification

### 3.4 Yield Sources Redesign (Critical Change)

**Problem**: BSC 的 ALP 本质是“对手方资金池” (Counterparty Vault)：**交易者亏损 + 交易费** 回流给 LP，往往在高波动时收益更高，从而对冲 LP 的 IL。
Sui 生态目前缺少同等成熟、可安全集成的 ALP/GLP 级别 vault。

**Reality Check (2026-03-05)**:
- **Typus TLP** 是最接近 ALP/GLP 的 GMX-style vault，但曾发生 **oracle exploit** → 视为 **HIGH RISK / opt-in**（仅作为未来可选引擎，不作为 v1 默认收益腿）
- 因此 v1 必须用更“可预测”的 carry 收益替代 ALP，并重新校准 LP 权重（详见 §9.5 PoC）

**Solution (v1)**: Replace ALP with **Lending carry + (optional) delta-hedged LST carry**, and treat CLMM as an opportunistic short-vol layer:

```
BSC Original                          Sui Replacement
============                          ===============
ALP (5-15% APY)          -->          USDC Lending (3-8% APY)
  trader losses = gains                 NAVI / Suilend / Scallop

                                      + (Optional) LST Carry (3-5% APY)
                                        afSUI / haSUI
                                        hedged by SUI perps when funding is favorable

V2 LP fees (5-20%)       -->          CLMM LP fees (5-30%+)
                                        Cetus concentrated liquidity
                                        (higher capital efficiency, but short-vol)
                                        gate position size by fee/vol edge (see §9.5)

1001x Delta Hedge         -->          Aftermath Perp Hedge (on-chain)
  cost: 1-5% funding                   cost: 1-5% funding
```

**Net Impact (updated)**:
- v1 的“保底收益”来自 lending carry（更稳但上限更低）
- CLMM 仍可带来上行空间，但它是 **short volatility**（短波动）暴露：需要动态权重与 range 策略，否则在高波动下可能出现负期望收益（PoC 已验证）

### 3.5 Flash Loan: Confirmed Available ✅

| Protocol | Flash Loan | Pattern | Fee |
|----------|-----------|---------|-----|
| NAVI Protocol | ✅ | Hot Potato (Receipt) | ~0.05% |
| Scallop | ✅ | Hot Potato (FlashTicket) | ~0.05% |
| DeepBook V3 | ✅ | Hot Potato (FlashLoan) | Free (0%) |

**Hot Potato Pattern** — Move's type system guarantees atomicity:

```
PTB (Programmable Transaction Block)
+-------------------------------------------------------------+
|  Step 1: borrow()  --> (Coin<USDC>, Receipt)                 |
|  Step 2: use funds (swap / rebalance / etc.)                 |
|  Step 3: repay(coin, receipt)  --> Receipt destroyed          |
|                                                               |
|  Receipt has ZERO abilities (can't drop/store/copy/key)      |
|  Not repaid in same PTB? --> Transaction REVERTS              |
|  Move type system enforces this — safer than Solidity!       |
+-------------------------------------------------------------+
```

### 3.6 Perp Hedge: Aftermath (Primary) + Bluefin (Optional)

- **Aftermath Perps (Primary)**:
  - Fully on-chain CLOB (orderbook + matching in Move contracts)
  - Supports USDC-collateralized markets (e.g. `BTC/USD:USDC`, `SUI/USD:USDC`)
  - Can be composed in PTB directly from Move (contract-callable)

- **Bluefin (Optional / Non-core)**:
  - API-based trading with off-chain orderbook; requires signed requests
  - Perps collateral may rely on bridged USDC (wUSDC) rather than native USDC
  - Therefore **not suitable** as the default on-chain hedge adapter for a fully autonomous vault

## 6. Key Flow Design

### 6.1 cycle() — 7-Phase State Machine

```
Phase 0: Pre-checks
  +-- elapsed >= min_cycle_interval?
  +-- Pyth price fresh (max_age <= 60s)?
  +-- Fail --> abort

Phase 1: Read State
  +-- VaultState totals
  +-- Adapter position values (CLMM, LST, Hedge)
  +-- Cash buffer = treasury.balance

Phase 2: TWAP + Regime
  +-- oracle::record_snapshot(pyth_price)
  +-- oracle::compute_regime()
  +-- Cold start (samples < 12) --> forced Normal
  +-- Regime change --> emit RegimeSwitched

Phase 3: Target Allocation
  +-- (yield%, lp%, buffer%) = get_allocation(&regime)
  +-- target_* = total_assets * pct / 10000
  +-- delta = current - target per strategy

Phase 4: Risk Control
  +-- TWAP vs Spot deviation > threshold --> OnlyUnwind
  +-- Hedge health < threshold --> OnlyUnwind
  +-- OnlyUnwind --> reduce only
  +-- N consecutive safe cycles --> restore Normal

Phase 5: Rebalance
  +-- OnlyUnwind: close_short, remove_liq, unstake only
  +-- Normal + small delta: rebalance_ptb()
  +-- Normal + large delta: rebalance_flash()

Phase 6: Process Queue
  +-- queue::process_queue()
  +-- Pending requests --> calculate USDC --> status = Ready

Phase 7: Bounty + Events
  +-- bounty = min(gas_est * markup, max_bps * assets)
  +-- Pay bounty to caller
  +-- Update last_cycle_ts
  +-- emit CycleExecuted
```

### 6.2 deposit()

```
deposit(usdc_coin)
  assert risk_mode != OnlyUnwind
  assert value > 0
  shares = (total_shares == 0) ? amount : amount * total_shares / total_assets
  mint Coin<SDYE>(shares) --> transfer to sender
  treasury += usdc
  total_assets += amount, total_shares += shares
  emit Deposit
```

### 6.3 withdraw()

```
request_withdraw(shares_coin)
  usdc = value(shares) * total_assets / total_shares
  if treasury >= usdc:
    instant redeem: burn shares, send usdc, emit InstantWithdraw
  else:
    queue: create WithdrawRequest, lock shares, emit WithdrawQueued

claim(request_id)
  assert status == Ready, sender == owner
  send usdc to owner, pay bounty to caller
  destroy WithdrawRequest, emit Claimed
```

## 7. Security Design

### 7.1 Sui-Specific Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-SUI-1 | Shared object congestion (DoS) | HIGH | Minimize shared objects, gas budget, degradation path |
| T-SUI-2 | External lib overflow (Cetus incident) | CRITICAL | Self-built math.move, validate all external returns |
| T-SUI-3 | Oracle manipulation | HIGH | TWAP + MIN_SAMPLES, Pyth freshness, deviation→OnlyUnwind |
| T-SUI-4 | MEV / Sandwich | MEDIUM | Slippage caps, PTB atomic, amount limits |
| T-SUI-5 | Clock time misuse (ms vs s) | MEDIUM | Uniform _ms naming, consistent units |
| T-SUI-6 | Dynamic field enumeration | LOW | Sequential IDs, no internal index exposure |
| T-SUI-7 | Excessive struct abilities | MEDIUM | Minimal abilities, no `drop` on share tokens |
| T-SUI-8 | Malicious package upgrade | HIGH | Immutable package or UpgradeCap + 48h timelock |

### 7.2 Access Control

| Function | Caller | Guard |
|----------|--------|-------|
| deposit() | Any user | risk_mode != OnlyUnwind |
| request_withdraw() | Any user | Holds Coin<SDYE> |
| claim() | Request owner | request.owner == sender |
| cycle() | Anyone | Time interval check |
| update_config() | Admin | Holds AdminCap |
| emergency_unwind() | Admin | Holds AdminCap |
| mint/burn_shares() | Internal | public(package) |

### 7.3 Invariants (8 Critical)

```
INV-1: total_shares == SUM(user SDYE) + queue.pending_shares     [Share conservation]
INV-2: total_assets >= treasury + yield + lp + hedge values      [Asset coverage]
INV-3: OnlyUnwind --> no new positions (reduce only)             [Risk constraint]
INV-4: WithdrawRequest shares cannot be double-claimed           [Redemption fairness]
INV-5: bounty <= max_bounty_bps * total_assets / 10000           [Bounty cap]
INV-6: cycle interval >= min_cycle_interval                      [Frequency limit]
INV-7: Flash loan repaid in same PTB (Hot Potato enforced)       [Atomicity]
INV-8: Cold start (samples < MIN_SAMPLES) forces Normal regime   [Safe startup]
```

### 7.4 Math Library Safety (Cetus Lessons)

```move
// All multiplications checked for overflow via u128 intermediate
fun safe_mul(a: u64, b: u64): u64 {
    let r = (a as u128) * (b as u128);
    assert!(r <= (U64_MAX as u128), EOverflow);
    (r as u64)
}

// All divisions checked for zero
fun safe_div(a: u64, b: u64): u64 {
    assert!(b > 0, EDivideByZero);
    a / b
}

// Critical paths use u128 intermediates
fun mul_div(a: u64, b: u64, c: u64): u64 {
    assert!(c > 0, EDivideByZero);
    let r = ((a as u128) * (b as u128)) / (c as u128);
    assert!(r <= (U64_MAX as u128), EOverflow);
    (r as u64)
}

const BPS_PRECISION: u64 = 10_000;
const PRICE_PRECISION: u64 = 1_000_000_000;
const SHARE_PRECISION: u64 = 1_000_000;
```

### 9.3 Regime Allocations (Sui)

**Conservative v1 Defaults (建议作为主网上线初始参数)**:

| Regime | Yield (Lending-first) | CLMM LP | Buffer | Notes |
|--------|------------------------|---------|--------|-------|
| CALM (vol<1%) | 70% | 27% | 3% | LP 仅在 edge>0 时增配；range 可更窄以提升 fee |
| NORMAL (1-3%) | 80% | 17% | 3% | 降低 short-vol 暴露；优先保证可赎回性 |
| STORM (>=3%) | 97% | 0% | 3% | LP 默认关闭；进入防守姿态，必要时 ONLY_UNWIND |

**Why change from the original 40/57/3?**
- BSC 版本依赖 ALP 的“高波动正相关收益”抵消 LP 风险；Sui v1 缺少同等成熟 ALP → 需要更保守的 LP 权重（见 §9.5 PoC）

### 9.4 Feasibility Check (Delta-Hedged CLMM)

对冲后的 CLMM LP 本质是“卖波动” (Short Vol / Short Gamma)，净收益近似：

```
NetCarry ≈ LP_Fees - GammaBleed(IL) - Funding - TradingCosts

Feasibility Condition:
  LP_Fee_APR_est  >  Funding_APR + HedgeTradeCost + LP_RecenterCost + SafetyMargin
```

因此 LP 不应该是固定权重，而应该是 **gate**：当 `edge <= 0` 时自动下调 LP 权重并回流到 lending。

- `poc/economics/sui_hedged_lp_sim.py`

命令与结果摘录（SUI 价格来自 CoinGecko 日线，模型假设见脚本头部注释）:


2) **Conservative v1 (Lending-first, LP capped, no LST, range ±30%)**: 小幅正收益（保底逻辑成立）


结论 (Conclusion):
- 在缺少 ALP 的情况下，**LP 权重 + range 策略**决定系统是否“正期望”
- v1 应以 lending 为底仓，CLMM 作为可选上行层，并在合约层引入 gate（按 fee/vol edge 自动调仓）

More frequent rebalancing = faster regime response = tighter risk control.
